Training comparison · 3 strategies × 3 models

🥊 Three training strategies, head-to-head

Same test set (Kaggle holdout — 1,022 patients, 50 stroke cases). Three different training mixes. Three model families. Threshold tuned on a validation split, then evaluated honestly.

🧪 The setup

Baseline

3,270 train rows · 159 positives

Kaggle only. Same data the original notebook used, but with class_weight='balanced' instead of SMOTE.

Variant A

3,988 train rows · 877 positives

Kaggle training set + all 898 NHANES stroke-positives. The "just add the new positives" approach.

Variant B

6,388 train rows · 877 positives

Same as A, plus 3,000 random NHANES negatives to dilute the dataset-origin signal.

The mitigations

  • The source column is never given to the model. Otherwise it would trivially learn "NHANES = stroke."
  • The test set is 100% Kaggle (1,022 patients, 50 positives) — the same population you actually care about predicting on.
  • Variant B adds NHANES negatives back so the model can't just learn population shape as a shortcut.
  • No SMOTE. class_weight='balanced' instead, which doesn't fabricate synthetic patients.
  • Thresholds tuned on a validation slice, not on the test set. F1-maximizing threshold from the PR curve.

🔬 The honest finding

Adding NHANES positives alone (Variant A) did NOT help — it actively hurt.

Compare Variant A against the Baseline:

  • RandomForest: F1 dropped from 0.258 → 0.170 (recall collapsed to 0.16)
  • GradBoost: F1 dropped from 0.279 → 0.152 (recall collapsed to 0.16)
  • LogReg: essentially unchanged (0.264 → 0.265)

This is exactly the selection-bias warning playing out in numbers. The tree models latched onto features that distinguish NHANES patients from Kaggle patients (subtle differences in age, glucose, BMI distributions), and that learning didn't transfer to the Kaggle-only test set. Linear models, which can't memorize population shape as easily, were unaffected.

Variant B (with NHANES negatives back) partially fixes it

Adding 3,000 random NHANES negatives forces the model to find actual stroke signal rather than population shape. RandomForest jumps from F1 0.170 → 0.303 (the best result overall). But the lift over Baseline is small — only +0.024 F1.

The takeaway in one line

The NHANES data is helpful but not transformative. The fundamental ceiling on this problem is set by Kaggle's tiny positive set (50 in the test fold) and the fact that demographic features just don't carry enough signal to distinguish strokes precisely. Precision of ~0.20 at recall ~0.5–0.7 looks like the realistic operating range — no matter how much you throw at it.

🎯 My recommendation

Ship Variant B + RandomForest

  • Operating point: threshold 0.252, Precision 0.193, Recall 0.700, F1 0.303
  • Interpretation: for every 100 patients the model flags, ~19 actually have stroke (vs ~5% base rate — so a flagged patient is roughly 4× more likely to be a true stroke than a random one). It catches 70% of strokes.
  • Why this and not the LR baseline: +21% precision at acceptable recall, more headroom for tuning.
  • Why not Variant A: the data showed it directly. NHANES positives without negatives = worse.

If you need higher precision, raise the threshold from 0.252 → ~0.5 and recall will drop to ~0.4 with precision ~0.30. There's no model trick to escape the precision-recall curve.
